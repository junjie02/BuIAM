# **1.** **方案综述**

本方案面向多 Agent 协作场景，构建一套独立于业务 Agent 系统之外的身份与访问控制控制面。系统以“身份可识别、能力可定义、委托可验证、请求可拦截、行为可追溯”为总体目标，围绕 Agent 身份认证、细粒度权限管控、委托授权链、审计追踪和原型演示闭环进行设计。

相较于传统围绕“用户—应用”关系建立的 IAM，本方案重点解决 User → Agent A → Agent B → Service C 这一调用链中的新型安全问题，包括身份混淆、权限扩散、信任链缺失和定责困难。方案采用“系统外独立控制面”思路，将安全能力从具体 Agent 框架中解耦，通过统一的身份服务、授权服务、委托服务、策略执行网关和审计中心，对多 Agent 协作过程实施强制控制。

在演示场景上，方案采用飞书企业协作原型：飞书文档助手 Agent 负责任务理解和报告生成，企业数据 Agent 是唯一可访问飞书企业数据的高权限 Agent，外部检索 Agent 仅能访问公开网页。系统将展示一条正常委托链和一条越权拦截链，并为两条链路输出完整审计日志。

# **2.** **建设背景与需求分析**

命题要求构建一个面向 AI Agent 的通用身份与权限系统，能够为每个 Agent 建立唯一、可校验的身份，支持能力声明管理、委托授权、越权拦截和审计追溯，并最终交付一个可运行的多 Agent 协作安全原型系统。

从业务本质上看，本项目不是传统后台权限系统的简单迁移，而是需要处理“主体不再只有用户，调用链不再只有单跳，执行者不再固定，任务上下文动态变化”的新型安全约束。系统必须能够回答以下问题：是谁发起了此次请求、当前是哪一个 Agent 在执行、它代表的是哪位用户、为何允许访问、为何拒绝访问，以及若发生安全事件应如何回溯到具体链路。

结合命题要求与评审关注点，本方案将需求归纳为六类：一是唯一身份与可信认证，二是覆盖主体、资源、动作、上下文的细粒度授权，三是可验证、可约束、可撤销的委托授权链，四是运行时越权拦截能力，五是全链路审计与任务可追踪能力，六是可运行、可演示、可复现的工程原型。

# **4.** **总体技术架构**

系统采用分层式总体架构，自上而下划分为交互接入层、多 Agent 业务层、安全控制面、资源工具层和数据与运维层。安全控制面以独立服务形态运行，对所有 Agent 调用进行统一认证、授权、委托校验与审计，不将安全逻辑散落在具体 Agent 代码中。

该架构的核心优势在于：第一，安全能力与具体 Agent 框架解耦，便于接入不同模型和推理引擎；第二，授权决策可以集中治理和版本化管理；第三，任何一跳 A2A 调用都必须经过统一网关，从而形成稳定的控制闭环。

## **4.1** **分层架构说明**

| **层级**        | **组成模块**                                                 | **职责说明**                                 |
| --------------- | ------------------------------------------------------------ | -------------------------------------------- |
| 交互接入层      | Web 控制台、CLI、演示脚本                                    | 接收用户任务、展示链路与日志、支持评委复现   |
| 多 Agent 业务层 | 文档助手 Agent、企业数据 Agent、外部检索 Agent               | 承担任务理解、数据查询、信息检索和结果汇聚   |
| 安全控制面      | Identity  Service、Token Service、Authorization Service、Delegation Service、Audit Service | 统一完成认证、授权、委托、校验、审计         |
| 策略执行层      | A2A Gateway /  Policy Enforcement Point                      | 所有 Agent 调用统一接入点，执行实时拦截      |
| 资源工具层      | 飞书 OpenAPI、企业知识库、外部网页检索接口                   | 作为受保护的目标资源和外部工具               |
| 数据与运维层    | PostgreSQL、Redis、日志存储、监控组件                        | 存储 Agent 信息、能力声明、委托链和缓存/日志 |

## **4.2** **逻辑架构说明**

一次典型请求的处理路径为：用户通过控制台发起任务，文档助手 Agent 解析任务并向安全控制面申请委托调用企业数据 Agent；控制面完成身份校验、权限交集计算和委托链签发后，A2A 网关放行调用；企业数据 Agent 在访问飞书资源前再次经过网关和授权服务验证；最终结果返回上游 Agent 进行汇总并写入文档，同时审计服务写入每一次允许或拒绝的决策日志。

在越权场景中，若外部检索 Agent 试图访问飞书企业数据，网关将在其调用企业数据 Agent 前触发授权检查。由于其能力声明不包含企业内部数据读权限，且不存在合法委托链，系统将立即返回明确拒绝码，并生成对应审计记录。

# **5.** **核心模块技术设计**

## **5.1 Agent** **身份与认证模块**

身份模块负责建立 Agent 的唯一身份标识，并为其签发访问令牌。每个 Agent 在系统中均拥有唯一 agent_id、显示名称、职责类型、可访问资源域、所属组织与状态标识。系统通过注册中心维护 Agent 元数据，通过 Token Service 按预设角色和能力范围签发 Access Token。

Token 的设计要求满足可签发、可校验、可过期、可吊销四项基本能力。所有下游服务不得信任调用方自带的明文字段，而必须通过控制面验签和解析 Token，确认其签发方、目标受众、有效期、主体身份和委托上下文后再做业务处理。

为提升安全性，方案建议令牌采用短时有效期，并支持 jti 唯一标识、黑名单吊销和来源绑定等能力。若比赛实现条件允许，可进一步在服务间增加 mTLS 或令牌与客户端实例绑定的约束，以降低令牌盗用风险。

## **5.2** **能力声明与细粒度授权模块**

能力声明模块用于定义 Agent 在静态层面上“能做什么、不能做什么”。一条能力声明至少应包括主体、资源类型、动作类型、资源范围、约束条件和状态等字段。例如，企业数据 Agent 具备读取飞书多维表格和通讯录的能力，外部检索 Agent 仅具备访问公开网页和摘要外部信息的能力。

授权模块在运行时结合请求上下文进行动态判断。其决策输入至少包括：当前主体、来源 Agent、目标资源、请求动作、代表用户、任务类型、时间窗口和委托链状态。系统遵循默认拒绝、最小授权、显式声明优先的原则；当请求未匹配到允许策略时一律拒绝。

授权引擎的输出不仅包含 allow 或 deny 结果，还应包括命中的策略标识、失败原因、建议错误码和审计上下文，以支撑错误提示与后续追踪。

## **5.3** **委托授权链模块**

委托授权链是本方案的核心创新模块，用于处理 Agent 代表用户向下游 Agent 发起受限调用的场景。系统不允许上游 Agent 直接转交自己的全部权限给下游 Agent，而必须由控制面根据“用户权限 ∩ 上游 Agent 能力 ∩ 下游 Agent 能力 ∩ 当前上下文约束”计算本次调用的有效权限，并签发一条受限的委托链记录。

每一跳委托记录至少包含 grant_id、parent_grant_id、task_id、delegated_user、source_agent、target_agent、scope、resource、action、issued_at、expires_at、status 和 delegation_depth 等字段。系统支持设置最大委托深度、防止权限扩张，并支持撤销上游委托后让所有下游链路自动失效。

在技术实现上，委托链服务分为三个步骤：创建根授权、签发子授权、校验链状态。根授权在用户首次触发任务时创建；子授权在 Agent 调 Agent 时按需签发；校验链状态在每次调用落地执行前完成，以确保委托没有过期、没有被撤销，也没有超出许可范围。

## **5.4** **策略执行网关与运行时拦截模块**

为避免安全逻辑被业务代码绕过，系统在所有 Agent 间调用前设置统一的策略执行网关。任何 Agent 若想请求其他 Agent、内部资源或外部工具，都必须先经过该网关。网关的职责包括解析令牌、提取调用上下文、调用授权服务、验证委托链、生成标准化错误响应以及写入审计日志。

网关需要处理三类典型拒绝：一是身份无效，如 Token 过期、签名错误、主体不存在；二是权限不足，如能力声明不包含当前资源或动作；三是委托非法，如调用者不具备再委托资格、委托深度超限、上游授权已撤销。

该设计使系统具备真正的运行时强制控制能力，也满足命题要求中“实时拦截越权行为”的核心验收点。

## **5.5** **审计追踪与任务链可视化模块**

审计模块负责记录每一次授权决策的完整上下文，既记录允许决策，也记录拒绝决策。审计日志不应只是简单的业务访问日志，而应包含 trace_id、task_id、request_id、user_id、source_agent_id、target_agent_id、resource、action、decision、reason_code、policy_id、token_id、delegation_id、created_at 等关键字段。

在查询侧，系统应支持按任务、按 Agent、按时间窗口、按结果类型等多条件检索，并支持导出日志文件。在可视化侧，可通过任务链视图或时序列表形式展示一条任务如何在多个 Agent 间流转、在哪一跳被放行或阻断。

审计模块是评委验收的重要抓手，也是系统可观测性的核心基础。

# **7.** **关键流程设计**

## **7.1** **正常委托流程**

• 用户在控制台输入“生成企业周报”，文档助手 Agent 接收任务并创建主任务 task_id。

• 文档助手 Agent 判断需要企业内部数据，向委托服务申请调用企业数据 Agent。

• 身份服务校验文档助手 Agent 的身份与令牌有效性。

• 授权服务检查用户权限、文档助手 Agent 能力、企业数据 Agent 能力以及上下文条件。

• 委托服务生成子授权 grant，并签发受限 Access Token。

• A2A 网关放行文档助手 Agent → 企业数据 Agent 调用。

• 企业数据 Agent 读取飞书多维表格并返回数据。

• 文档助手 Agent 汇总结果并写入飞书文档。

• 审计服务为以上每个关键节点写入 allow 日志。

## **7.2** **越权拦截流程**

• 外部检索 Agent 尝试请求企业数据 Agent 读取飞书多维表格。

• 网关首先校验外部检索 Agent 的令牌与身份状态。

• 授权服务发现其 Capability Statement 不包含企业内部数据访问能力。

• 同时系统未查询到合法委托链，因此权限交集计算结果为空。

• 网关返回明确错误码，例如 AUTHZ_DENY_CAPABILITY_MISMATCH。

• 审计服务记录一条 deny 日志，标记来源 Agent、目标资源、拒绝原因和策略编号。

## **7.3** **异常与边界流程**

系统应对以下异常场景做出清晰响应：Token 过期、令牌被吊销、委托深度超限、上游授权已失效、目标 Agent 超时、飞书接口失败、外部检索服务不可用等。对权限类异常，系统统一返回语义化错误与审计记录；对业务类异常，主 Agent 需能够感知失败并向用户给出降级或重试提示。

在可选增强设计上，系统可进一步实现 Token 实时撤销、过期自动清理、风险等级升高时的人工审批、以及对 Prompt Injection 场景下高危工具调用的二次确认。

# **9.** **安全设计要点**

本方案围绕五项安全基线展开：第一，默认拒绝，所有未显式授权的请求均不得通过；第二，最小权限，任何下游授权都只能在上游权限子集内收缩；第三，委托可验证，委托链中的每一跳都必须可查询、可校验、可撤销；第四，控制与执行分离，业务 Agent 不拥有最终的权限解释权；第五，全链路留痕，允许与拒绝均保留审计上下文。

针对 Access Token 盗用风险，方案建议使用短时令牌、受众限制、jti 黑名单、来源绑定和服务端实时校验等措施。针对 Prompt Injection 等 AI 特有风险，方案强调“模型意图不等于执行权限”，任何对敏感资源和工具的实际调用均需重新经过网关和授权服务判断，而不能因为上游模型给出调用指令就自动放行。

针对高权限 Agent 风险，系统通过能力声明和委托链将高权限访问收敛到企业数据 Agent 一类受控节点，不允许普通 Agent 直接接触内部数据资源，从而减少权限面暴露。

# **13.** **方案亮点与预期成效**

本方案的核心亮点在于将多 Agent IAM 设计成系统外独立控制面，而不是某个 Agent 框架内部的权限子模块。通过这一设计，系统能够跨框架复用统一的身份、授权、委托与审计能力，从而更贴近企业级平台化落地路径。

方案的第二个亮点是把“用户权限 ∩ Agent 能力 ∩ 下游能力 ∩ 上下文约束”的交集逻辑显式建模为委托授权链，使委托过程可解释、可验证、可撤销，而不是将委托简化为一条普通字段传递。

方案的第三个亮点是将允许和拒绝两类授权决策都纳入统一审计，使评委能够清楚看到任务如何在多个 Agent 间流转、系统为何放行、又为何拦截，显著提升演示可读性与验收通过率。

# **14.** **结论**

综上，本技术方案围绕命题要求提出了一套结构清晰、实现路径明确、可复现演示的多 Agent 身份与权限系统设计。该方案不仅满足 Agent 身份认证、细粒度授权、委托授权、越权拦截和审计追溯等核心功能要求，还通过独立控制面架构提升了系统的通用性、扩展性和工程可信度，适合作为比赛项目的正式技术方案底稿。
